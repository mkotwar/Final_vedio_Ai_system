# Event Aggregator Forensic Audit

## Objective
Analyze the current state of `app/services/event_aggregation.py` to document its operational logic, identify architectural weaknesses, and determine compatibility with the new structured metadata (`FrameRichMetadata`).

## 1. Architecture Diagram & Event Creation Flow

**Current Event Aggregation Flow**
```text
Frames Metadata (Sorted by Time)
        ↓
Continuity Scoring
(Jaccard similarity on tokens/strings with 1-frame tolerance)
        ↓
Group Frames into Temporal Clusters
        ↓
Feature Extraction
(Most common activities, deduplicated objects by string match)
        ↓
Unified Text Generation
(Concatenate captions, keywords, activities, object descriptions)
        ↓
Heuristic Analysis
(Regex/Keyword checks on unified text for Event Type & Behavioral Flags)
        ↓
Primary Agent Identification
(Hardcoded string priority: Person > Vehicle > Animal)
        ↓
Event Dictionary Construction & JSON Export
```

## 3. Heuristic Inventory

### Text Similarity
*   **SequenceMatcher**: Used in `calculate_similarity` on `caption` and `scene_type`.
*   **Jaccard Similarity**: Used extensively in `calculate_event_continuity_score`. Frames are tokenized, stop-words removed, and sets are compared to yield `Actor`, `Activity`, `Scene`, and `Behavior` scores.

### Keyword Matching
*   **Entity Identification**: `PERSON_TYPES = {"person", "pedestrian", "guard", ...}`, `VEHICLE_TYPES = {"vehicle", "car", "truck", ...}` are used to determine if an object is an actor.
*   **Event Inferencing**: `infer_event_type` relies exclusively on substring matching. Example: `if any(kw in text for kw in ["robbery", "theft", "stealing"]) -> return "robbery_incident"`.
*   **Behavioral Flags**: Substring checks in `acts_lower` (e.g., `if "running" in acts_lower: flags.append("person_running")`).

### Caption-Based Logic
*   **Unified Text**: `group_unified_text` concatenates `captions`, `keywords`, `activities`, and `objects_desc`. This megastring becomes the sole source of truth for the heuristic keyword matchers.

## 4. Current Weakness Analysis

### Weakness A: Caption Dependency
**Evidence:** `unified_text` is generated by concatenating captions and keywords. If the VLM omits an action in the `activities` list but includes it in the `caption`, the aggregator regex might still trigger it. Conversely, if `caption` and `keywords` are removed (as in V3 Minimal Schema), the `unified_text` string drastically shrinks, potentially causing all heuristic keyword matches to fail.

### Weakness B: Regex & String Matching Dependency
**Evidence:** The core logic inside `infer_event_type` and `_compute_behavioral_flags` heavily relies on hardcoded string lists (e.g., `["weapon", "gun", "knife"]`). This is highly fragile. If a VLM uses a synonym not in the list (e.g., `"firearm"` vs `"gun"`), the aggregator will miss the event entirely.

### Weakness C: No Actor Tracking
**Evidence:** Objects are deduplicated by checking `key = (t.lower(), st.lower(), col.lower())` rather than using the structured `id` (e.g., `person_1`). Primary agents are selected simply by taking the first matched person in the list. The aggregator is blind to which specific actor is performing an action.

### Weakness D: No Relationship Awareness
**Evidence:** The structured `relationships` array is not consumed anywhere in `process_events`. The aggregator attempts to infer interactions by string matching the `unified_text` (e.g., `if "robbery" in text`), ignoring the exact semantic graph provided by the VLM (e.g., `person_1 -> facing -> person_2`).

### Weakness E: No Spatial Reasoning
**Evidence:** The `location_context` array is completely ignored. Instead, location is crudely extracted from OCR text using regex (`re.search(r'(Gate\s+\d+\s+Enterance\s+Area)', location_text)`). If OCR fails or the environment is indoors, the location defaults to `"the monitored area"`.

## 5. Structured Metadata Compatibility Analysis

The current aggregator is **incompatible** with structured metadata.
*   `activities`: Currently consumed, but reduced to tokens and substring matches.
*   `relationships`: Ignored. **Should enter the pipeline** to explicitly link `subject_id` to `target_id` to build an interaction graph.
*   `location_context`: Ignored. **Should enter the pipeline** to replace the OCR fallback logic and provide semantic spatial positioning (e.g., `near_counter`).
*   `objects`: Consumed, but `id` fields (repaired heavily by Phase 0.5) are ignored.

## 6. Current vs Target Architecture Mapping

**Current Aggregator Architecture**
```text
Frame Metadata (Captions, Keywords, Strings)
        ↓
Continuity Score (Jaccard token similarity)
        ↓
Grouped Frames
        ↓
String Concatenation (Unified Text)
        ↓
Regex / Hardcoded Keyword Matching
        ↓
String-based Event Typing & Primary Actor Guessing
        ↓
Event JSON Output
```

**Target Aggregator Architecture (Conceptual)**
```text
Structured Metadata (Objects[id], Activities, Relationships, Location Context)
        ↓
Actor Graph Tracking (Persistent identity via IDs: person_1, person_2)
        ↓
Interaction Graph Parsing (Mapping subject_id -> relation -> target_id)
        ↓
Semantic Continuity Grouping (Frames grouped by graph topology, not string similarity)
        ↓
Event Timeline Construction
        ↓
Narrative Event Generation (Driven by structured interactions, not Regex)
```
